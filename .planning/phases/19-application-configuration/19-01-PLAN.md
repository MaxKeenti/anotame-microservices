<objective>
Externalize identity-service and catalog-service datasource URL and HTTP port configuration so they can be fully controlled by environment variables. Remove hardcoded `quarkus.datasource.jdbc.url=jdbc:postgresql://anotame-db:5432/anotame` from both files, replace with `%dev` profile fallbacks pointing to their dedicated per-service local containers, and change the hardcoded port to `${PORT:808x}`.

Purpose: Railway injects QUARKUS_DATASOURCE_JDBC_URL and PORT at deploy time. With the hardcoded URL present, Railway's env var wins due to ordinal priority (300 > 250), but if the env var is ever absent the service silently connects to the wrong host. The intent is explicit failure if QUARKUS_DATASOURCE_JDBC_URL is unset in prod. The %dev profile provides safe local dev fallback without that risk.

Output: Two modified application.properties files — identity-service and catalog-service — with CFG-01, CFG-02, and CFG-03 verified for both services.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| application.properties → production runtime | Committed config file visible in VCS; must not contain prod credentials or reachable prod host strings |
| env var → Quarkus runtime | Railway injects QUARKUS_DATASOURCE_JDBC_URL, QUARKUS_DATASOURCE_USERNAME, QUARKUS_DATASOURCE_PASSWORD at ordinal 300; absence of base URL in config enforces that prod cannot start without these vars |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19-01 | Information Disclosure | application.properties (jdbc.url) | mitigate | Remove `quarkus.datasource.jdbc.url=jdbc:postgresql://anotame-db:5432/anotame` from both files; no prod-reachable host string committed to VCS |
| T-19-02 | Information Disclosure | application.properties (credentials) | accept | Credentials already use `${QUARKUS_DATASOURCE_USERNAME:admin}` / `${QUARKUS_DATASOURCE_PASSWORD:password}` — dev-only fallbacks are weak by design; prod values injected via env vars never touch the file |
| T-19-03 | Denial of Service | quarkus.http.port (hardcoded) | mitigate | Change to `${PORT:8082}` — if PORT is absent in prod the fallback port still works; Railway's port assignment is always honored when set |
| T-19-04 | Spoofing | %dev.quarkus.datasource.jdbc.url fallback | accept | Dev fallback points to localhost only; cannot be reached from outside the developer's machine; no sensitive data in dev DB |
</threat_model>
