# Phase 2 Plan 02: JWT Key Migration to Environment Variables

**Phase:** 2
**Plan:** 02
**Goal:** Remove file-path JWT key references from all four services and deliver keys via environment variables so Railway builds succeed without committed PEM files.
**Req IDs:** SEC-02

---

## Overview

All four services currently use `mp.jwt.verify.publickey.location=publicKey.pem` (file-path variant). Identity-service also uses `smallrye.jwt.sign.key.location=privateKey.pem`. The PEM files exist locally but are gitignored and never committed — so Railway CI builds fail silently (or the developer manually places files in the Docker build context). This plan migrates to inline env var delivery: remove the `.location` property variants, add the inline property variants that read from `MP_JWT_VERIFY_PUBLICKEY` and `SMALLRYE_JWT_SIGN_KEY`, generate a fresh RSA-2048 key pair, populate `.env` for local dev, and produce a `DEPLOY_NOTES.md` with the Railway rollout sequence.

**CRITICAL deployment order** (wrong order causes live auth failures):
1. Set env vars in Railway for ALL 4 services first
2. Deploy all 4 services simultaneously (or within the same Railway release)
3. Never deploy `application.properties` changes before Railway env vars are set

## Tasks

### Task 1: Generate a new RSA-2048 key pair

**File**: `identity-service/src/main/resources/privateKey.pem`, `identity-service/src/main/resources/publicKey.pem` (local only — gitignored)

**Action**: Run from the project root:

```bash
cd anotame-api/backend/identity-service/src/main/resources
openssl genrsa -out privateKey.pem 2048
openssl rsa -in privateKey.pem -pubout -out publicKey.pem
```

Confirm the files were generated:
```bash
ls -la anotame-api/backend/identity-service/src/main/resources/*.pem
```

Both files must exist. Confirm git does NOT track them:
```bash
git ls-files anotame-api/backend/identity-service/src/main/resources/privateKey.pem
git ls-files anotame-api/backend/identity-service/src/main/resources/publicKey.pem
```

Both commands must return empty (no output).

Now populate the JWT key entries in `.env` that were left blank in plan 02-01. The full PEM content — including the `-----BEGIN...-----` and `-----END...-----` header lines — must be stored as a single env var value. In `.env` files, multi-line values must be wrapped in double quotes.

Update `.env` to replace the blank `SMALLRYE_JWT_SIGN_KEY=` and `MP_JWT_VERIFY_PUBLICKEY=` lines with the actual key content:

```bash
# Export the keys to shell variables first to inspect them
PRIVATE_KEY=$(cat anotame-api/backend/identity-service/src/main/resources/privateKey.pem)
PUBLIC_KEY=$(cat anotame-api/backend/identity-service/src/main/resources/publicKey.pem)
echo "Private key length: ${#PRIVATE_KEY}"
echo "Public key length: ${#PUBLIC_KEY}"
```

Then manually edit `.env` to set:

```
SMALLRYE_JWT_SIGN_KEY="-----BEGIN RSA PRIVATE KEY-----
MIIEpAIBAAKCAQEA...
(full key content, all lines)
...
-----END RSA PRIVATE KEY-----"

MP_JWT_VERIFY_PUBLICKEY="-----BEGIN PUBLIC KEY-----
MIIBIjANBgkqhkiG9...
(full key content, all lines)
...
-----END PUBLIC KEY-----"
```

The exact PEM content will differ since keys are generated fresh. Copy the verbatim output of `cat privateKey.pem` and `cat publicKey.pem` into `.env`, wrapping each in double quotes.

**Verify**: `grep -c "BEGIN RSA PRIVATE KEY" .env` returns `1`. `grep -c "BEGIN PUBLIC KEY" .env` returns `1`. `git ls-files .env` returns empty (still untracked — confirmed by plan 02-01).

### Task 2: Update identity-service `application.properties`

**File**: `anotame-api/backend/identity-service/src/main/resources/application.properties`

**Action**: Edit the JWT Configuration section. The current content (lines 23-29) is:

```properties
# JWT Configuration (Using generated keys for dev)
mp.jwt.verify.issuer=anotame-identity
smallrye.jwt.sign.key.location=privateKey.pem
mp.jwt.verify.publickey.location=publicKey.pem
quarkus.smallrye-jwt.enabled=true
mp.jwt.token.header=Cookie
mp.jwt.token.cookie=jwt
```

Replace it with:

```properties
# JWT Configuration
# Keys are delivered via environment variables (not file paths).
# SMALLRYE_JWT_SIGN_KEY  -> smallrye.jwt.sign.key  (private key, identity-service only)
# MP_JWT_VERIFY_PUBLICKEY -> mp.jwt.verify.publickey (public key, all services)
mp.jwt.verify.issuer=anotame-identity
smallrye.jwt.sign.key=${SMALLRYE_JWT_SIGN_KEY}
mp.jwt.verify.publickey=${MP_JWT_VERIFY_PUBLICKEY}
quarkus.smallrye-jwt.enabled=true
mp.jwt.token.header=Cookie
mp.jwt.token.cookie=jwt
```

Key changes:
- `smallrye.jwt.sign.key.location=privateKey.pem` REMOVED — replaced by `smallrye.jwt.sign.key=${SMALLRYE_JWT_SIGN_KEY}`
- `mp.jwt.verify.publickey.location=publicKey.pem` REMOVED — replaced by `mp.jwt.verify.publickey=${MP_JWT_VERIFY_PUBLICKEY}`
- The property names `smallrye.jwt.sign.key` and `mp.jwt.verify.publickey` (no `.location` suffix) are the inline content variants — this is intentional and different from the `.location` variants.

Do not modify any other section of this file.

**Verify**: `grep "location" anotame-api/backend/identity-service/src/main/resources/application.properties` returns empty (no `.location` properties remain). `grep "SMALLRYE_JWT_SIGN_KEY" anotame-api/backend/identity-service/src/main/resources/application.properties` returns a match.

### Task 4: Create `DEPLOY_NOTES.md` with Railway rollout sequence

**File**: `.planning/phases/02-security-foundations/DEPLOY_NOTES.md`

**Action**: Create the file with the following exact content:

```markdown
# SEC-02 Railway Deployment Notes

**Date prepared**: 2026-03-31
**Risk**: HIGH — wrong deployment order causes live JWT verification failures

## Why order matters

After this plan's `application.properties` changes are deployed, services will read the JWT
public key from the `MP_JWT_VERIFY_PUBLICKEY` environment variable. If that env var is not set
in Railway when the service starts, SmallRye JWT will fail to initialize and ALL authenticated
requests will return 401 or 500.

Similarly, identity-service reads `SMALLRYE_JWT_SIGN_KEY` to sign tokens. If not set, login
returns 500.

## Safe rollout sequence

1. **Generate key pair locally** (done in Task 1 of plan 02-02)

2. **Set Railway env vars BEFORE deploying any service**

   For `identity-service` in Railway dashboard (or via `railway variables set`):
   - `SMALLRYE_JWT_SIGN_KEY` = full content of `privateKey.pem` (multi-line PEM)
   - `MP_JWT_VERIFY_PUBLICKEY` = full content of `publicKey.pem` (multi-line PEM)

   For `catalog-service`, `sales-service`, `operations-service` in Railway:
   - `MP_JWT_VERIFY_PUBLICKEY` = same public key as above (identical value)

3. **Verify env vars are set in Railway before proceeding**

   Railway CLI check:
   ```bash
   railway variables --service identity-service | grep JWT
   railway variables --service catalog-service | grep JWT
   railway variables --service sales-service | grep JWT
   railway variables --service operations-service | grep JWT
   ```
   Each must show `SMALLRYE_JWT_SIGN_KEY` (identity only) and `MP_JWT_VERIFY_PUBLICKEY`.

4. **Deploy all 4 services in the same Railway release** (trigger redeploy simultaneously)

   Do NOT stagger. A brief window where identity-service has new keys but catalog-service
   still has old keys (or no env var) will cause token verification failures for active users.

5. **Post-deploy smoke test**

   ```bash
   # Login — should return 200 and Set-Cookie: jwt=...
   curl -X POST https://<identity-service-url>/auth/login \
     -H "Content-Type: application/json" \
     -d '{"username":"<test-user>","password":"<test-password>"}'

   # Use the returned JWT cookie to call a protected endpoint
   curl https://<catalog-service-url>/catalog/services \
     -H "Cookie: jwt=<token>"
   ```

## Rollback plan

If deployment fails:
1. Revert `application.properties` changes in all 4 services (re-add `.location` variants)
2. Ensure `publicKey.pem` is present in the Docker build context for each service
3. Deploy again

The old PEM-file approach is safe to roll back to because the files were never committed
and the new key pair is local-only until Railway env vars are confirmed.
```

**Verify**: File exists at `.planning/phases/02-security-foundations/DEPLOY_NOTES.md`. `wc -l .planning/phases/02-security-foundations/DEPLOY_NOTES.md` returns at least 50 lines.
