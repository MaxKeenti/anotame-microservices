<!-- Staging validate approach: local Docker PostgreSQL — no Railway staging service required. -->

Flyway CLI is not on developer PATH. Use the official Flyway Docker image for validate:
```bash
docker run --rm \
  --network host \
  flyway/flyway:latest \
  -url=jdbc:postgresql://localhost:5433/anotame_staging \
  -user=admin \
  -password=admin \
  -locations=filesystem:/migrations \
  validate
```

Alternatively, validate can be inferred from service startup: if all 4 services start without Flyway errors after the schema restore, validate passes implicitly (Flyway runs validate internally before migrate).

Staging DB specs:
- Container name: anotame-staging
- Port: 5433 (avoids conflict with dev anotame-db on 5432)
- DB name: anotame_staging
- User: admin / Password: admin

The schema dump from plan 06-02 (/tmp/anotame_v1_baseline.sql) is the source for staging restore. If the dump was generated to a different path, adjust accordingly.

Service startup for staging validation:
- All 4 services must be started with QUARKUS_DATASOURCE_URL pointing to the staging DB
- Use %prod profile to test the production Flyway behavior (baseline-on-migrate, migrate-at-start, database.generation=none)
- docker compose up --build with overridden DB connection is the most practical approach

<task type="auto">
  <name>Task 2: Start all 4 services against staging DB and verify Flyway history tables</name>
  <files>
    docker-compose.yml
  </files>
  <action>
Start all 4 backend services against the staging DB using environment variable overrides. This tests the exact production Flyway behavior without touching the live Railway DB.

**Option A — docker compose with env overrides (recommended):**
```bash
STAGING_URL="jdbc:postgresql://host.docker.internal:5433/anotame_staging"

cd "/Users/maximilianogonzalezcalzada/Library/Mobile Documents/com~apple~CloudDocs/source/personal/anotame-microservices"

QUARKUS_DATASOURCE_JDBC_URL="${STAGING_URL}" \
QUARKUS_DATASOURCE_USERNAME=admin \
QUARKUS_DATASOURCE_PASSWORD=admin \
QUARKUS_PROFILE=prod \
docker compose up --build identity-service catalog-service sales-service operations-service 2>&1 | tee /tmp/staging_startup.log
```

Note: `QUARKUS_PROFILE=prod` activates the `%prod` profile, which enables `%prod.quarkus.hibernate-orm.database.generation=none` — this tests that Flyway takes ownership of DDL in production mode.

**Watch startup logs for success indicators:**
- GOOD: "Successfully applied N migration(s) to schema" or "Successfully validated N migration(s)"
- GOOD: "Flyway migration has completed" for each service
- BAD: "Found non-empty schema(s) ... but no schema history table" — means baseline-on-migrate is not working
- BAD: "Migration checksum mismatch" — means V1 baseline file was modified after being applied
- BAD: "HHH90000025" — Hibernate auto-DDL warning, means %prod profile not activating

After all 4 services have started (or failed), check history tables:
```bash
docker exec anotame-staging psql -U admin -d anotame_staging -c "\dt flyway_schema_history_*"
```
Expected: 4 rows — flyway_schema_history_identity, flyway_schema_history_catalog, flyway_schema_history_sales, flyway_schema_history_operations.

Stop services after verification:
```bash
docker compose down
```
