pg_dump not on developer PATH. Two fallback approaches:
1. Railway CLI tunnel: `railway connect {service}` opens a local port tunnel to the Railway PostgreSQL instance; use local pg_dump against that port.
2. Docker exec: if the local anotame-db container is running WITH all 4 services having been started against it at least once (so auto-DDL ran), use: `docker exec anotame-db pg_dump -U admin -d anotame --schema-only`
   WARNING: A freshly seeded Docker DB (only init.sql run, services never started) is MISSING the auto-DDL tables. Do NOT use a fresh container.

The pg_dump approach produces ONE full-schema SQL file. All 4 services receive the same V1 content (full schema). This is safe because:
- baseline-version=1 means Flyway never tries to execute V1 — it only reads its checksum for validate
- Each service owns its own history table, so having the full schema in V1 does not cause cross-service conflicts
- The single-file approach avoids complex per-table scoping that risks missing auto-DDL columns

Quarkus Flyway default migration location: classpath:db/migration

  <name>Task 2: Generate V1__baseline.sql from live Railway database</name>
  <what-needed>
The V1 baseline must be generated from the live Railway PostgreSQL database using pg_dump --schema-only. This requires a human action because the developer must authenticate with Railway to access the production DB.

The live DB contains auto-DDL-created tables (tco_work_order, tco_work_order_item) and columns (daily_capacity_minutes on tce_establishment) that do not exist in init.sql. These MUST be captured in V1.

WARNING: Do NOT run pg_dump against a freshly-seeded Docker container that has never had all 4 services started against it — the auto-DDL tables will be absent.
  </what-needed>
  <instructions>
Choose ONE of the two approaches below and run the pg_dump command. Then copy the output SQL file to all 4 service migration directories as V1__baseline.sql.

**Approach A — Railway CLI tunnel (preferred if Railway CLI is installed):**
```bash
# Step 1: Link to the Railway project if not already linked
railway link

# Step 2: Open a tunnel to the production PostgreSQL service
# This will print a connection string including the local port
railway connect anotame-db
# Tunnel stays open in this terminal — keep it running

# Step 3: In a NEW terminal, run pg_dump against the tunnel port
# Replace <PORT> with the port shown by `railway connect` (typically 5432 or a random local port)
pg_dump -h localhost -p <PORT> -U admin -d anotame --schema-only -f /tmp/anotame_v1_baseline.sql

# Step 4: Close the tunnel (Ctrl+C in the first terminal)
```

**Approach B — Docker exec (if local anotame-db container has been fully bootstrapped):**
```bash
# Prerequisite: All 4 services must have been started at least once against this container
# so that Hibernate auto-DDL has run and created tco_work_order, tco_work_order_item, daily_capacity_minutes

# Check that the auto-DDL tables exist before dumping:
docker exec anotame-db psql -U admin -d anotame -c "\dt tco_work_order"
# If this returns "Did not find any relation named tco_work_order" — use Approach A instead.

# If tables exist, dump:
docker exec anotame-db pg_dump -U admin -d anotame --schema-only > /tmp/anotame_v1_baseline.sql
```

**After generating /tmp/anotame_v1_baseline.sql, copy it to all 4 services:**
