<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App service → PostgreSQL (Railway private network) | JDBC connections travel over Railway's private .railway.internal network — not public internet |
| Public internet → /q/health/ready | Health endpoint is publicly reachable (Railway exposes it for health checks); returns only UP/DOWN status |
| Railway dashboard → env vars | Datasource credentials are entered in Railway's encrypted variable store; they are never in the git repository |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-08 | Information Disclosure | /q/health/ready endpoint | accept | Quarkus health endpoint returns {"status":"UP"} and service name. No PII, no credentials, no stack traces. This is the standard Railway health check integration pattern. |
| T-20-09 | Elevation of Privilege | Railway env var template variable resolution | mitigate | Template variables `${{identity-db.PGHOST}}` are resolved by Railway at deploy time — only services in the same project can reference each other's variables. Cross-project access is not possible. Verify in Railway dashboard that each service only references its own DB (not another service's DB). |
| T-20-10 | Information Disclosure | Railway public URL exposure | accept | Railway assigns public HTTPS URLs to each service. The only publicly accessible endpoint in this phase is /q/health/ready. All other API endpoints require JWT authentication (Phase 2 — already implemented). |
| T-20-11 | Denial of Service | Flyway migration on fresh PostgreSQL | accept | First deploy runs Flyway V1 baseline migration on empty DB. This is a one-time operation. Subsequent deploys skip migration if schema is current (Flyway checksum validation). Migration failure would cause service startup failure — visible in Railway logs. |
| T-20-12 | Spoofing | Railway PostgreSQL PGHOST resolution | accept | PGHOST resolves to internal `.railway.internal` hostname. Private network traffic is not accessible outside the Railway project. No additional TLS configuration needed for same-project private networking. |
</threat_model>
