<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Build stage → Run stage | Multi-stage Docker build: only quarkus-app/ artifacts cross the boundary; Maven toolchain and credentials stay in build stage |
| Railway runner → container registry | Railway builds the image from checked-in Dockerfile; no external secrets in Dockerfile |
| Runtime container → PostgreSQL | JDBC credentials injected via Railway env vars at runtime (Phase 19 + Phase 20 Wave 3) — not baked into image |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-01 | Information Disclosure | Dockerfile ENV layer | accept | MAVEN_OPTS contains only JVM tuning flags (-Xmx512m, --add-opens) — no secrets. Docker layer inspection reveals only public configuration. |
| T-20-02 | Tampering | go-offline → dependency:resolve replacement | mitigate | dependency:resolve downloads artifacts from Maven Central at build time; Railway builds run in isolated ephemeral containers; no MITM risk beyond Railway's own build infrastructure trust. Use `-B` (batch mode) to prevent interactive prompts. |
| T-20-03 | Elevation of Privilege | Runtime ENTRYPOINT system property | accept | `-Djava.util.logging.manager=org.jboss.logmanager.LogManager` is a standard Quarkus logging configuration property. It does not grant elevated privileges or change security manager configuration. |
| T-20-04 | Information Disclosure | Docker image layers | accept | Build stage artifacts (Maven local repo, source code) are not copied to run stage. Multi-stage build ensures only quarkus-app/ (compiled application) is in the final image. No source code or credentials in published image. |
</threat_model>
