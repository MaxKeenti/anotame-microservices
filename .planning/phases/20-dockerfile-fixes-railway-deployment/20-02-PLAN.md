<objective>
Create the 4 `railway.toml` configuration files (one per service) that tell Railway to use the Dockerfile builder with the correct path, health check endpoint, and timeout. Also delete the legacy GHCR build pipeline files (`build_and_push.sh` and `anotame-db/`).

Purpose: Railway's auto-detect would choose Nixpacks/Railpack if no railway.toml is present. An explicit `builder = "dockerfile"` ensures deterministic, correct builds. The health check configuration (300s timeout for Quarkus augmentation startup) prevents Railway from killing the container prematurely. The legacy files are now fully replaced by Railway native builds.

Output: 4 railway.toml files committed; build_and_push.sh and anotame-db/ removed from git history via git rm.
</objective>

Minimal railway.toml structure (D-06 — no restartPolicy, no startCommand override):
```toml
[build]
builder = "dockerfile"
dockerfilePath = "{service-name}/Dockerfile"

[deploy]
healthcheckPath = "/q/health/ready"
healthcheckTimeout = 300
```

Railway Root Directory for ALL 4 services: `anotame-api/backend`
This means dockerfilePath is relative to that root.

dockerfilePath per service:
- identity-service: "identity-service/Dockerfile"
- catalog-service:  "catalog-service/Dockerfile"
- sales-service:    "sales-service/Dockerfile"
- operations-service: "operations-service/Dockerfile"

File location per service (relative to repo root):
- anotame-api/backend/identity-service/railway.toml
- anotame-api/backend/catalog-service/railway.toml
- anotame-api/backend/sales-service/railway.toml
- anotame-api/backend/operations-service/railway.toml

IMPORTANT — monorepo pitfall from RESEARCH.md:
Railway's Config File Path does NOT automatically follow the Root Directory setting.
After committing these files, the executor (Wave 3 human) must set Config File Path
in Railway dashboard for each service to the absolute path:
  /anotame-api/backend/identity-service/railway.toml  (etc.)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| railway.toml → Railway build system | railway.toml is a checked-in config file Railway reads to determine build behavior; it controls which Dockerfile is used |
| git rm → repository history | Deletion of anotame-db/ removes init.sql from working tree; git history preserves it for recovery |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-05 | Tampering | railway.toml builder field | mitigate | `builder = "dockerfile"` is an explicit opt-in to Dockerfile builds. If this value is changed to "nixpacks" or omitted, Railway falls back to auto-detection. The file is version-controlled — any tampering is visible in git history. |
| T-20-06 | Information Disclosure | anotame-db/init.sql deletion | accept | The init.sql file is deleted from working tree but remains in git history. It contains DDL only (no credentials, no data). Git history is not published to a public remote, so this is acceptable. |
| T-20-07 | Denial of Service | healthcheckTimeout too low | accept | 300s is intentionally generous. If reduced below ~120s, Railway would kill Quarkus containers on cold start. The value is locked in railway.toml and visible in code review. |
</threat_model>
