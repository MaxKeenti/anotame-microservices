# Phase 3 Plan 01: JWT Claim Enrichment (identity-service)

**Phase:** 3
**Plan:** 01
**Goal:** Add `user_id` and `branch_id` custom claims to every JWT issued by identity-service so that downstream services can read verified user identity without trusting unvalidated headers.
**Req IDs:** DATA-01, DATA-03 (prerequisite)
**Depends on:** none

---

## Overview

`JwtUtils.generateToken()` currently signs only `upn` (username) and `groups` (roles). Neither the real user UUID nor the user's assigned branch UUID is present in the token. This plan adds both claims by (1) creating a native-query helper to look up the active branch assignment at login time, (2) updating `JwtUtils.generateToken()` to accept `userId` and `branchId`, and (3) updating every call site in `AuthService` (`login`, `register` via delegation, `updateCredentials`).

## Tasks

### Task 1: Add `findActiveBranchForUser` native query to `UserRepository`

**File**: `anotame-api/backend/identity-service/src/main/java/com/anotame/identity/infrastructure/persistence/repository/UserRepository.java`

**Action**: Edit

Add the following import and method to `UserRepository`. Do NOT create a new entity class for `tce_employee_assignment` — use `EntityManager` directly to avoid building an entity hierarchy for a cross-context join table.

Add import at the top of the file (after the existing imports):

```java
import jakarta.inject.Inject;
import jakarta.persistence.EntityManager;
import jakarta.persistence.NoResultException;
```

Add field injection inside the class (after the class declaration, before existing methods):

```java
@Inject
EntityManager em;
```

Add method at the end of the class (before the closing `}`):

```java
/**
 * Returns the active branch UUID for the given user, or null if no active
 * assignment exists. Uses a native query to avoid introducing a full entity
 * for the cross-context join table tce_employee_assignment.
 */
public UUID findActiveBranchForUser(UUID userId) {
    try {
        Object result = em.createNativeQuery(
                "SELECT id_branch FROM tce_employee_assignment " +
                "WHERE id_user = :userId AND is_active = true LIMIT 1")
                .setParameter("userId", userId)
                .getSingleResult();
        return UUID.fromString(result.toString());
    } catch (NoResultException e) {
        return null;
    }
}
```

### Task 2: Update `JwtUtils.generateToken()` to include `user_id` and `branch_id` claims

**File**: `anotame-api/backend/identity-service/src/main/java/com/anotame/identity/infrastructure/security/JwtUtils.java`

**Action**: Edit

Add `java.util.UUID` import. Change the method signature and body as shown below.

Before (current file, lines 16-22):

```java
public String generateToken(String username, Set<String> roles) {
    return Jwt.issuer(issuer)
            .upn(username)
            .groups(roles)
            .expiresIn(Duration.ofHours(24))
            .sign();
}
```

After:

```java
/**
 * @param username  The user's login name (used as upn).
 * @param userId    The user's real UUID from tca_user.id_user. Always present.
 * @param branchId  The user's active branch UUID from tce_employee_assignment.
 *                  May be null for users with no active branch assignment
 *                  (e.g., newly registered users). When null the claim is
 *                  omitted from the token; sales-service uses its rollout
 *                  fallback in that case.
 * @param roles     Role codes (e.g. {"EMPLOYEE"}).
 */
public String generateToken(String username, java.util.UUID userId,
                             java.util.UUID branchId, Set<String> roles) {
    var builder = Jwt.issuer(issuer)
            .upn(username)
            .claim("user_id", userId.toString())
            .groups(roles)
            .expiresIn(Duration.ofHours(24));
    if (branchId != null) {
        builder = builder.claim("branch_id", branchId.toString());
    }
    return builder.sign();
}
```

**Verify**: File compiles — `mvn compile -pl anotame-api/backend/identity-service -q`. Expect a compile error in `AuthService` (call sites not yet updated) — that is resolved in Task 3.
