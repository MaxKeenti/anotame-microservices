<objective>
Rewrite docker-compose.yml to contain only 4 independent PostgreSQL containers (one per service) on distinct host ports 5431–5434. Remove all Quarkus app containers, the frontend container, the shared anotame-db container, the init.sql volume mount, the anotame-network bridge, and the shared postgres_data volume. Keep pgadmin for developer convenience. Also prune .env of vars that were only relevant to the removed app containers.

Purpose: Developers run `quarkus:dev` on the host (not via Docker) and need isolated per-service databases matching the port assignments Phase 19 wires into each service's `%dev` profile. The current compose file is designed for production-mode Docker startup; after this plan, it is a clean "databases only" file that stays running in the background while developers work.

Output: A rewritten docker-compose.yml with 4 DB containers + pgadmin, and a pruned .env retaining only JWT keys and developer-relevant vars.

DEV-02 dependency note: DEV-02 (each service's %dev profile points to its own container) is jointly satisfied by this plan (containers provisioned on correct ports) and Phase 19 (which writes the %dev datasource URLs into each application.properties). If Phase 19 has not run before this plan executes, the containers will exist but services will still connect to anotame-db on first quarkus:dev start — run Phase 19 to complete DEV-02.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| docker-compose.yml → DB containers | Compose file is committed to VCS; credentials embedded in it are dev-only (admin/password) |
| .env → host quarkus:dev process | .env contains JWT private/public keys; must not be committed with real prod keys |
| DB containers → host network | Containers bind to localhost ports (127.0.0.1 implicit via Docker Desktop on macOS); not exposed to network interfaces |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-01 | Information Disclosure | docker-compose.yml POSTGRES_PASSWORD | accept | Value is `password` — a known dev-only credential with no prod surface; DB containers bind to localhost only on macOS Docker Desktop; acceptable for local dev |
| T-21-02 | Information Disclosure | .env SMALLRYE_JWT_SIGN_KEY | accept | The key in .env is a dev-only RSA key pair; prod keys are injected via Railway env vars and never touch this file; .env is not committed to a public repo |
| T-21-03 | Elevation of Privilege | Per-service volume isolation | mitigate | Each service gets its own named volume (identity_db_data, catalog_db_data, etc.) — blast radius of one compromised container is limited to its own volume; no shared volume between services |
| T-21-04 | Information Disclosure | Healthcheck command | accept | `pg_isready -U admin -d identity` does not expose table data or credentials in output; standard postgres healthcheck pattern |
| T-21-05 | Denial of Service | catalog-db host port 5432 conflict | accept | Port 5432 is the PostgreSQL default and may conflict with a locally-installed PostgreSQL server; documented in README (Plan 02) and in a compose comment; developer responsibility to stop local PG |
</threat_model>
