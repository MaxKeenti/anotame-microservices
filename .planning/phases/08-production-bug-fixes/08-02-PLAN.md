From anotame-web/src/lib/services/api.svelte.ts:
The `apiService.request()` method throws `new Error(backendMessage)` for non-OK responses.
It does NOT attach a `status` property to the thrown Error object — it only throws `new Error(string)`.
To detect 409, the executor MUST NOT check `e.status` (it will be undefined).
Instead, detect 409 by parsing the response text. However, `apiService.request` already consumed the response body before throwing.

The reliable detection strategy: catch the error thrown by `apiService.request`, then check whether the original fetch returned 409. Since `apiService.request` re-throws `Error(backendMessage)` for all non-OK statuses, the only safe approach is to NOT use `apiService.request` for the DELETE call, OR to detect 409 from the error message text.

RECOMMENDED approach (simplest, no API surface change): Call `fetch` directly for the DELETE in `handleCancel` / `handleCancelWorkOrder`, check `response.status === 409` before throwing, and if 409, show the specific toast. Example skeleton:

```ts
const response = await fetch(`${API_SALES}/orders/${order.id}`, {
  method: 'DELETE',
  credentials: 'include',
  headers: { 'Content-Type': 'application/json' }
});
if (response.status === 409) {
  toast.error('No se puede eliminar', {
    description: 'El pedido tiene órdenes de trabajo asociadas. Elimina las órdenes de trabajo primero.'
  });
  return;
}
if (!response.ok) {
  toast.error('Error al cancelar pedido');
  return;
}
```

ALTERNATIVE approach (no raw fetch): Check `e.message` for a 409 signature from the backend. The backend throws a message like "API Error: 409 Conflict" (since `backendMessage = \`API Error: ${response.status} ${response.statusText}\`` when there is no JSON body). So:
```ts
} catch (e: any) {
  if (e?.message?.includes('409')) {
    toast.error('No se puede eliminar', { description: '...' });
  } else {
    toast.error('Error al cancelar pedido', { description: e?.message });
  }
}
```

Use whichever approach is cleaner given the existing code. The ALTERNATIVE (message check) requires no new imports and keeps the pattern consistent with existing handlers in the file — prefer it.
