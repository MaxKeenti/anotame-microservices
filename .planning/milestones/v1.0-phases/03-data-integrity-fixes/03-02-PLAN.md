# Phase 3 Plan 02: Ticket Number Sequence (DB + sales-service)

**Phase:** 3
**Plan:** 02
**Goal:** Replace the collision-prone `System.currentTimeMillis() % 10000` ticket number with a PostgreSQL sequence that guarantees uniqueness under any concurrency.
**Req IDs:** DATA-02
**Depends on:** none

---

## Overview

`tco_order.ticket_number` has a `UNIQUE` constraint but is currently populated with `"ORD-" + System.currentTimeMillis() % 10000`, which wraps every ~10 seconds and produces duplicates under concurrent load. The fix is a PostgreSQL sequence (`tco_ticket_number_seq`) applied via explicit DDL (Hibernate `ddl-auto=update` will NOT create sequences), a new `nextTicketNumber()` method added to the `OrderRepositoryPort` domain port, an implementation in `OrderPersistenceAdapter` using a native query, and replacement of the broken line in `SalesService`. The `folio_branch` field — currently hardcoded to `1` — is set to the same sequence value (cast to `int`) so both fields stay consistent.

## Tasks

### Task 1: Apply sequence DDL to the database

**File**: `anotame-api/backend/sales-service/src/main/resources/db/sequence-migration.sql` (new file — for documentation and repeatable execution)

**Action**: Create

Create this file with the exact DDL:

```sql
-- Phase 3 DATA-02: collision-free ticket number sequence
-- Apply BEFORE deploying the updated sales-service.
-- IF NOT EXISTS makes this idempotent (safe to re-run).
CREATE SEQUENCE IF NOT EXISTS tco_ticket_number_seq
    START WITH 1
    INCREMENT BY 1
    NO CYCLE;
```

Then apply it to the running database:

```bash
docker compose exec postgres psql -U postgres -d anotame_db \
  -c "CREATE SEQUENCE IF NOT EXISTS tco_ticket_number_seq START WITH 1 INCREMENT BY 1 NO CYCLE;"
```

Confirm the sequence was created:

```bash
docker compose exec postgres psql -U postgres -d anotame_db \
  -c "SELECT sequencename FROM pg_sequences WHERE sequencename = 'tco_ticket_number_seq';"
```

Expected output: one row with `tco_ticket_number_seq`.

**Verify**: The `SELECT` above returns exactly one row. No error on the `CREATE SEQUENCE` statement (existing sequence is a no-op due to `IF NOT EXISTS`).

### Task 2: Add `nextTicketNumber()` to `OrderRepositoryPort` and implement it in `OrderPersistenceAdapter`

**File 1**: `anotame-api/backend/sales-service/src/main/java/com/anotame/sales/application/port/output/OrderRepositoryPort.java`

**Action**: Edit

Add the following method signature to the interface (after the existing `void delete(...)` method, before the KPI metrics comment block):

```java
/**
 * Returns the next ticket number from the PostgreSQL sequence tco_ticket_number_seq.
 * Format: ORD-00001, ORD-00042, etc. (zero-padded to 5 digits).
 * This is the ONLY correct way to generate ticket numbers — do NOT use
 * System.currentTimeMillis() or UUID-based approaches, both of which collide.
 */
String nextTicketNumber();
```

**File 2**: `anotame-api/backend/sales-service/src/main/java/com/anotame/sales/infrastructure/persistence/adapter/OrderPersistenceAdapter.java`

**Action**: Edit

Add `EntityManager` injection (add after the existing `@Inject OrderRepository orderRepository;` field):

```java
@Inject
jakarta.persistence.EntityManager em;
```

Add the `nextTicketNumber()` implementation (add after the `getDailyWorkload` override, before the `private Order toDomain` method):

```java
@Override
public String nextTicketNumber() {
    Long next = ((Number) em.createNativeQuery(
            "SELECT nextval('tco_ticket_number_seq')").getSingleResult()).longValue();
    return String.format("ORD-%05d", next);
}
```

Note: `getSingleResult()` returns a `BigInteger` from the PostgreSQL JDBC driver; casting via `Number` and calling `.longValue()` handles both `Long` and `BigInteger` safely.

**Verify**: Module compiles — `mvn compile -pl anotame-api/backend/sales-service -q`.

### Task 3: Replace the broken ticket-number and folio-branch lines in `SalesService`

**File**: `anotame-api/backend/sales-service/src/main/java/com/anotame/sales/application/service/SalesService.java`

**Action**: Edit

Replace lines 43-44 (current hardcoded values):

Before:
```java
order.setTicketNumber("ORD-" + System.currentTimeMillis() % 10000); // Simple ID generation

```

After:
```java
String ticketNumber = orderRepository.nextTicketNumber();
order.setTicketNumber(ticketNumber);
```

## Success Verification

Maps to phase success criteria:

- **Criteria 2 (ticketNumber no collisions)**: `tco_ticket_number_seq` is a server-side sequence; PostgreSQL guarantees each call to `nextval()` returns a strictly increasing, non-repeating value even under concurrent connections. The `UNIQUE` constraint on `ticket_number` will never be violated by this approach.
- **Criteria 4 (existing orders unaffected)**: The sequence starts at 1 and only applies to new inserts. Existing rows with their current `ticket_number` values are untouched. `CREATE SEQUENCE IF NOT EXISTS` is idempotent.
