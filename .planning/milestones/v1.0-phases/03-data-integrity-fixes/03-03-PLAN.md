# Phase 3 Plan 03: JWT-Sourced branchId and createdBy (sales-service)

**Phase:** 3
**Plan:** 03
**Goal:** Replace the hardcoded branch UUID and username-hash `createdBy` in `SalesService` with verified values read from the JWT, and remove the untrustworthy `X-User-Name` header.
**Req IDs:** DATA-01, DATA-03
**Depends on:** 03-01 (identity-service must be deployed so JWTs carry `user_id` and `branch_id` before this plan goes to production; the code change is safe to write in parallel but must not be deployed first)

---

## Overview

`OrdersResource.createOrder()` currently trusts an `X-User-Name` HTTP header that any caller can forge. `SalesService.createOrder()` then derives `createdBy` from a deterministic UUID hash of that string, and hardcodes `branchId`. This plan switches the controller to inject `JsonWebToken jwt`, extracts the verified `user_id` and `branch_id` claims, and passes real `UUID` values down to `SalesService`. The service signature changes from `(request, String username)` to `(request, UUID userId, UUID branchId)`. A rollout fallback is kept for `branch_id` to handle in-flight tokens that predate Plan 03-01's deployment.

## Tasks

### Task 1: Inject `JsonWebToken` into `OrdersResource` and extract claims

**File**: `anotame-api/backend/sales-service/src/main/java/com/anotame/sales/infrastructure/web/controller/OrdersResource.java`

**Action**: Edit

**Add import** at the top of the file (after existing imports):

```java
import org.eclipse.microprofile.jwt.JsonWebToken;
```

**Add field injection** inside the class, after `@Inject SalesService salesService;`:

```java
@Inject
JsonWebToken jwt;
```

**Replace `createOrder` method** (current lines 25-28):

Before:
```java
@POST
public Order createOrder(@jakarta.validation.Valid CreateOrderRequest request, @HeaderParam("X-User-Name") String username) {
    return salesService.createOrder(request, username);
}
```

After:
```java
@POST
public Order createOrder(@jakarta.validation.Valid CreateOrderRequest request) {
    UUID userId = UUID.fromString((String) jwt.getClaim("user_id"));

    // TODO: remove fallback after all sessions refreshed following 03-01 deployment
    String branchClaim = jwt.getClaim("branch_id");
    UUID branchId = (branchClaim != null)
            ? UUID.fromString(branchClaim)
            : UUID.fromString("ea22f4a4-5504-43d9-92f9-30cc17b234d1");

    return salesService.createOrder(request, userId, branchId);
}
```

The `@HeaderParam("X-User-Name") String username` parameter is removed entirely. The JWT is guaranteed to be valid at this point because `OrdersResource` is `@Authenticated` — SmallRye JWT rejects unsigned or expired tokens before the method is invoked.

Do NOT add `@RequestScoped` to `OrdersResource`. `jwt.getClaim()` called at method invocation time is safe in `@ApplicationScoped` beans — it reads from the current request context. Field-level `@Claim` injection (which would require `@RequestScoped`) is not used here.

**Verify**: File compiles — `mvn compile -pl anotame-api/backend/sales-service -q`. Expect a compile error in `SalesService` until Task 2 is complete.

After:

**Replace lines 45-46** (the hardcoded branchId and hashed createdBy):

Before:

## Success Verification

Maps to phase success criteria:

- **Criteria 1 (branchId = JWT branch_id)**: `OrdersResource` reads `jwt.getClaim("branch_id")` (added by Plan 03-01) and passes the parsed `UUID` directly to `SalesService`. The hardcoded UUID literal `ea22f4a4-...` exists only as a rollout fallback for in-flight old tokens; it is removed once 03-01 has been live long enough for all sessions to refresh.
- **Criteria 3 (createdBy = JWT user_id)**: `jwt.getClaim("user_id")` returns the real `tca_user.id_user` UUID (set by Plan 03-01). `UUID.nameUUIDFromBytes` is gone.
- **Criteria 4 (existing orders unaffected)**: Only new inserts are affected. Existing `tco_order` rows keep their current `id_branch` and `created_by_user_id` values. The rollout fallback ensures no insert fails due to a missing `branch_id` claim during the transition window.
