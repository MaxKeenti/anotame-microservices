# Phase 2 Plan 04: Cookie Secure Flag + PgAdmin Password Externalization

**Phase:** 2
**Plan:** 04
**Goal:** Gate the auth cookie's `secure` flag to the `%prod` profile and move the PgAdmin admin password out of `docker-compose.yml` into `.env`.
**Req IDs:** SEC-05, SEC-06

---

## Overview

Two small but concrete security gaps remain. First, `anotame.auth.cookie.secure=false` is hardcoded with no profile gating — production cookies are sent without the `Secure` attribute, making them transmissible over HTTP. Adding `%prod.anotame.auth.cookie.secure=true` to `application.properties` fixes this using MicroProfile Config's profile prefix mechanism. Second, `docker-compose.yml` hardcodes `PGADMIN_DEFAULT_PASSWORD: password` on the `anotame-pgadmin` service, which has no `env_file` directive unlike every other service. Both changes are straightforward edits with no code changes required.

---

## Tasks

### Task 1: Add `%prod` profile gate to cookie secure flag

**File**: `anotame-api/backend/identity-service/src/main/resources/application.properties`

**Action**: Locate the Auth Cookie Configuration section (currently lines 37-39):

```properties
# Auth Cookie Configuration
anotame.auth.cookie.secure=false
anotame.auth.cookie.same-site=Lax
```

Change it to:

```properties
# Auth Cookie Configuration
# secure=false for local dev (HTTP). %prod overrides to true for Railway (HTTPS).
anotame.auth.cookie.secure=false
%prod.anotame.auth.cookie.secure=true
anotame.auth.cookie.same-site=Lax
```

The `%prod.` prefix causes Quarkus to apply `anotame.auth.cookie.secure=true` only when the active profile is `prod` (the default Railway deployment profile). The base value `false` remains active for `%dev` (local docker-compose).

Do not change `anotame.auth.cookie.same-site=Lax`. `SameSite=Lax` is correct for both environments given the SvelteKit BFF server-side proxy topology.

The env var `ANOTAME_AUTH_COOKIE_SECURE=false` in `.env` provides a docker-compose override that maps to the same property via MicroProfile Config env var convention. This is redundant with the properties file base value but harmless — leave it in `.env` as-is (it confirms dev intent explicitly).

**Verify**: `grep -n "anotame.auth.cookie.secure" anotame-api/backend/identity-service/src/main/resources/application.properties` must return 2 lines: the base `false` line and the `%prod.` line.

Build to confirm no syntax error:
```bash
cd anotame-api/backend && ./mvnw compile -pl identity-service -am -q
```
Must exit with code 0.

### Task 2: Externalize PgAdmin password in `docker-compose.yml`

**File**: `docker-compose.yml`

**Action**: Locate the `anotame-pgadmin` service definition (currently lines 32-45):

```yaml
  anotame-pgadmin:
    image: dpage/pgadmin4:8
    container_name: anotame-pgadmin
    restart: always
    environment:
      PGADMIN_DEFAULT_EMAIL: admin@anotame.com
      PGADMIN_DEFAULT_PASSWORD: password
    ports:
      - "5050:80"
    networks:
      - anotame-network
    depends_on:
      anotame-db:
        condition: service_healthy
```

Change it to:

```yaml
  anotame-pgadmin:
    image: dpage/pgadmin4:8
    container_name: anotame-pgadmin
    restart: always
    env_file:
      - .env
    environment:
      PGADMIN_DEFAULT_EMAIL: admin@anotame.com
      PGADMIN_DEFAULT_PASSWORD: ${PGADMIN_DEFAULT_PASSWORD}
    ports:
      - "5050:80"
    networks:
      - anotame-network
    depends_on:
      anotame-db:
        condition: service_healthy
```

Two changes:
1. Add `env_file: - .env` (same pattern used by all other services in this file)
2. Change `PGADMIN_DEFAULT_PASSWORD: password` to `PGADMIN_DEFAULT_PASSWORD: ${PGADMIN_DEFAULT_PASSWORD}`

The `PGADMIN_DEFAULT_PASSWORD=password` value was added to `.env` in plan 02-01 Task 3. Docker Compose resolves `${PGADMIN_DEFAULT_PASSWORD}` from the `env_file` contents at startup.

**Verify**: `grep -A 3 "PGADMIN_DEFAULT_PASSWORD" docker-compose.yml` must show `${PGADMIN_DEFAULT_PASSWORD}` (not the literal `password`). `grep -B 1 "PGADMIN_DEFAULT_PASSWORD" docker-compose.yml` must show `env_file:` in the vicinity (confirming the directive was added). Run `docker compose config` to validate YAML syntax:
```bash
docker compose config --quiet
```
Must exit with code 0.

### Task 3: Smoke test both changes locally

**Action**: Start the full stack to verify PgAdmin starts correctly and the cookie flag behaves as expected:

```bash
docker compose up --build -d
docker compose ps
```

All services must show status `running` or `healthy`. Wait 30 seconds for services to initialize.

**Test PgAdmin password externalization:**
```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:5050
```
Expected: `200` or `302` (PgAdmin login page loads — not 500 or connection refused).

Manually open `http://localhost:5050` in a browser and log in with:
- Email: `admin@anotame.com`
- Password: `password` (the value from `.env`)

PgAdmin must accept these credentials.

**Test cookie secure flag (dev profile — must be false):**
```bash
curl -v -X POST http://localhost:8081/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"password"}' 2>&1 | grep "Set-Cookie"
```
Expected output must contain `Set-Cookie: jwt=...` WITHOUT the word `Secure`. For example:
```
< Set-Cookie: jwt=eyJ...; Path=/; HttpOnly; SameSite=Lax
```
The absence of `Secure` confirms the `%dev` profile is active and `anotame.auth.cookie.secure=false` is in effect.

The `%prod` secure=true behavior is verified in Railway (HTTPS environment) after deployment — not locally.

**Verify**: PgAdmin accessible with credentials from `.env`. Dev login response `Set-Cookie` header does NOT contain `Secure`.
