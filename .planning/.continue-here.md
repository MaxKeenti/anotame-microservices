## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|----------------------|
| Anchoring on memory-SIGKILL hypothesis | "Connection refused + restart loop + silent container death" pattern matched cgroup OOM-kill perfectly, but was also consistent with Railway edge routing to the wrong upstream port (Railway kills containers that fail healthcheck, which also leaves no log trail). Memory fix was applied before the actual root cause (Railway domain Target Port set to 8082/8083/8084 while containers listen on 8080) was identified. | advisory | For any "connection refused from cloud edge" bug, verify upstream port config in the dashboard BEFORE investigating container internals. Edge-level 502s can be routing misconfig just as easily as container death. |

<current_state>
Production 502 outage on catalog, sales, and operations services has been **root-caused but NOT yet resolved in production**. Identity service is healthy.

**Root cause (confirmed via Railway support chat):** Railway public-domain Target Port for catalog/sales/operations was configured to 8082/8083/8084 (matching the `${PORT:NNNN}` fallbacks that each service had in `application.properties`), but all containers actually listen on 8080 (Railway injects `PORT=8080` which overrides the fallback). Every edge request therefore hit an unlistened port → "connection refused" → Railway killed the "unhealthy" container → restart loop → no logs (SIGKILL flushes nothing).

Identity survives because its Railway Target Port happens to match what the container listens on.

**What's already done (committed):**
- Dockerfile ENTRYPOINT memory hardening on all 4 services: `-XX:MaxRAMPercentage=60.0 -XX:InitialRAMPercentage=50.0 -XX:MaxMetaspaceSize=192m -Xss512k -XX:+ExitOnOutOfMemoryError` (commits d0d4879 / 69d50fe / c96134c). Not the root cause, but good defensive instrumentation — `+ExitOnOutOfMemoryError` will make any future real OOM loud instead of silent.

**What's done on disk but NOT committed:**
- All 4 `application.properties` fallback ports changed from `${PORT:8081|8082|8083|8084}` → `${PORT:8080}`. This does not change prod behavior (Railway's injected PORT still wins), but it aligns what Railway's auto-port-detection would read going forward. dev.sh passes `-Dquarkus.http.port=<N>` at JVM level, so local dev is unaffected by this change.
</current_state>
