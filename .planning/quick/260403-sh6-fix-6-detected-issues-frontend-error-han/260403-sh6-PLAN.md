Create new file src/lib/services/ApiError.ts with custom error class exposing HTTP status code:

```typescript
export class ApiError extends Error {
  constructor(
    message: string,
    public status: number,
    public body?: unknown
  ) {
    super(message);
    this.name = 'ApiError';
  }
}

export function isApiError(error: unknown): error is ApiError {
  return error instanceof ApiError;
}
```

This provides type-safe error handling for all API errors with status code accessible to catch blocks.

In SalesService.java, locate the createOrder() and updateOrder() methods.

**Current problem:** Methods use `.toLocalDateTime()` which discards timezone offset information.

**Fix:** Replace all instances of `.toLocalDateTime()` with timezone-aware handling:

1. For `createdAt` and `updatedAt` fields, use: `OffsetDateTime.now(ZoneId.systemDefault())`
   - Or if you prefer UTC normalization: `Instant.now()` and persist as `TIMESTAMP WITH TIME ZONE`
2. Check the Order entity field types - if they're defined as `OffsetDateTime`, keep them as-is
3. If Order entity uses `LocalDateTime`, update the entity definition to `OffsetDateTime` and ensure the database column is `TIMESTAMP WITH TIME ZONE`

The key is: timezone information must be preserved from creation through save, not truncated to LocalDateTime which loses offset context.

Example for createOrder:
- Before: `order.setCreatedAt(LocalDateTime.now().toLocalDateTime())`
- After: `order.setCreatedAt(OffsetDateTime.now(ZoneId.systemDefault()))`

Same fix applies to updateOrder method.
