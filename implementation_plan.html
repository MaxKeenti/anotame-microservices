Implementation Plan

Capacitor Integration for the Anotame Staff App

Add Capacitor to anotame-web so the existing SvelteKit dashboard can be packaged as native iOS and Android apps for shop staff — without breaking the Railway-deployed web build.

01Context

The original markdown plan proposed a dual-build strategy (keep adapter-node for Docker, add adapter-static for Capacitor). After a grilling session that walked the design tree, several load-bearing gaps in the original draft surfaced — addressed here.

Blocker

HttpOnly cookie auth is structurally incompatible with a cross-origin Capacitor WebView.

auth.svelte.ts stores user state and relies on the backend's jwt cookie; apiService sends credentials: "include". In a Capacitor WebView the origin is capacitor://localhost (iOS) or https://localhost (Android). WKWebView's ITP and SameSite rules will silently drop the cookie. Native must move to Bearer-token + secure storage.

Drift

Env-var mechanism mismatch. The codebase reads backend URLs via $env/dynamic/public (runtime). The original plan used Vite's import.meta.env.PUBLIC_*. Under adapter-static there is no Node runtime, so only $env/static/public (baked at build time) works consistently.

Drift

The proposed SVELTE_CONFIG=svelte.config.mobile.js switch isn't real. SvelteKit always loads svelte.config.js. The original plan's build:mobile would silently produce the wrong (Node) adapter output.

Tradeoff

Setting ssr = false in a shared +layout.ts would regress the Docker web build (theme FOUC, weaker admin guard, broken %lang% substitution). The fix is a build-time file swap that scopes ssr = false to the mobile bundle only.

Strategy

Apple App Store guideline 4.2 rejects pure WebView wrappers. The plan now ships Camera (garment intake photos) and Biometric login as v1 native capabilities. Push notifications and barcode scanning are deferred.

02Decisions

Q1
Target audience: Staff Member (the person operating the shop). The mobile bundle wraps the existing dashboard; no new customer surface is built.
Q2
Auth model: Bearer token in the Authorization header, stored via @capacitor/preferences, for native. Web stays on HttpOnly cookies. Backend must return the JWT in the login response body — currently AuthController.createCookieResponse drops it.
Q3
Env strategy: Backend URLs baked at build time via $env/static/public. The mobile build runs on dev/CI (not Railway), so the pipeline mirrors Railway's env values locally.
Q4
Build switch: One svelte.config.js, branching on process.env.MOBILE === '1'. cross-env added for Windows.
Q5
SSR scope: ssr = false applies only to the mobile bundle, via a script-based file swap of +layout.mobile.ts over +layout.ts during build:mobile.
Q6
Locale: Paraglide's client runtime; static index.html ships with lang="es" and strings swap on hydration based on the PARAGLIDE_LOCALE cookie.
Q7
CORS: All four Quarkus services move cors.origins to an env-driven ANOTAME_CORS_ORIGINS property, defaulting to dev origins plus Capacitor schemes.
Q8
Native capabilities (v1): Camera (garment intake) and Biometric login. Push notifications and barcode scanner deferred.
Q9
Wrap-up: Add ADR-0005; delete the redundant routes/api/[...path]/+server.ts proxy; verify Capacitor version via npm view before pinning.

03Backend Changes (Quarkus)

B-A1 · Return JWT in login response body

File: anotame-api/backend/identity-service/src/main/java/com/anotame/identity/infrastructure/web/controller/AuthController.java

Replace the body of createCookieResponse so it returns the full AuthResponse:

return Response.ok(authResponse).cookie(cookie).build();

Web still uses the HttpOnly cookie (it ignores the body's token). Native reads and stores it. No new endpoint needed.

B-A2 · Env-driven CORS on all four services

Files: identity-service, catalog-service, sales-service, operations-service — each application.properties.

Replace the literal origins line with:

quarkus.http.cors.origins=${ANOTAME_CORS_ORIGINS:http://localhost:3000,capacitor://localhost,https://localhost,http://localhost}

Set ANOTAME_CORS_ORIGINS per environment in Railway (prod web origin + Capacitor schemes; staging adds the staging web origin). The default covers local Docker plus the Capacitor WebView origins.

B-B2 · Dual-adapter — svelte.config.js

One config, branched on the build env:

import nodeAdapter from '@sveltejs/adapter-node';
import staticAdapter from '@sveltejs/adapter-static';

const isMobile = process.env.MOBILE === '1';

const config = {
  kit: {
    adapter: isMobile
      ? staticAdapter({ pages: 'build', assets: 'build', fallback: 'index.html', strict: false })
      : nodeAdapter(),
    paths: { relative: false }
  },
  vitePlugin: {
    dynamicCompileOptions: ({ filename }) =>
      filename.includes('node_modules') ? undefined : { runes: true }
  }
};

export default config;

B-B3 · Capacitor project config — capacitor.config.ts new

import type { CapacitorConfig } from '@capacitor/cli';

const config: CapacitorConfig = {
  appId: 'com.anotame.staff',
  appName: 'Anotame Staff',
  webDir: 'build'
};

export default config;

Bundle ID can be revised during execution; com.anotame.staff reflects the audience decision (Q1) and leaves room for a future com.anotame.customer.

B-B4 · Runtime detection + API base — src/lib/capacitor.ts new

import { Capacitor } from '@capacitor/core';
import {
  PUBLIC_IDENTITY_URL,
  PUBLIC_CATALOG_URL,
  PUBLIC_SALES_URL,
  PUBLIC_OPERATIONS_URL
} from '$env/static/public';

export function isNativeApp(): boolean {
  return Capacitor.isNativePlatform();
}

const NATIVE_BASES = {
  identity: PUBLIC_IDENTITY_URL,
  catalog: PUBLIC_CATALOG_URL,
  sales: PUBLIC_SALES_URL,
  operations: PUBLIC_OPERATIONS_URL
} as const;

export function getApiBase(service: keyof typeof NATIVE_BASES): string {
  return isNativeApp() ? NATIVE_BASES[service] : `/api/${service}`;
}

Web never reads NATIVE_BASES (the isNativeApp() branch is always false), so empty values in the web build's .env are fine.

B-B5 · Native token store — src/lib/services/native/tokenStore.ts new

import { Preferences } from '@capacitor/preferences';

const KEY = 'anotame_jwt';

export const tokenStore = {
  async get():   Promise<string | null> { const { value } = await Preferences.get({ key: KEY }); return value; },
  async set(token: string): Promise<void> { await Preferences.set({ key: KEY, value: token }); },
  async clear():            Promise<void> { await Preferences.remove({ key: KEY }); }
};

B-B6 · Platform-aware ApiService — src/lib/services/api.svelte.ts modify

  • Replace the four hard-coded /api/... constants with getApiBase(...).
  • In request(): when native, read tokenStore.get() and set Authorization: Bearer ${token}; omit credentials: 'include'.
  • Web path: untouched.

B-B7 · Auth service hook into token store — src/lib/services/auth.svelte.ts modify

  • After successful login(), on native: await tokenStore.set(response.token).
  • In logout() and clearLocalSession(), on native: await tokenStore.clear().
  • setLocaleCookie works unchanged — WebViews accept same-origin cookie writes.

B-B8 · Convert server loads to universal loads rename

  • src/routes/(app)/+layout.server.ts → +layout.ts · LayoutServerLoad → LayoutLoad. The theme fetch already used SvelteKit fetch; only the file name and import type change.
  • src/routes/(app)/dashboard/admin/+layout.server.ts → +layout.ts · same conversion. Backend already enforces ADMIN role on every protected endpoint — this layout was defense-in-depth only.

B-B9 · Mobile-only SSR override — src/routes/+layout.mobile.ts new

export const ssr = false;
export const prerender = false;

Plus two Node scripts:

  • scripts/swap-mobile-layout.mjs — copies +layout.mobile.ts → +layout.ts before vite build.
  • scripts/restore-mobile-layout.mjs — removes the generated +layout.ts after the build.

The runtime +layout.ts is gitignored.

B-B10 · Locale placeholder in static build

When MOBILE=1, replace %lang% in build/index.html with es (ADR-0003 designates Spanish as authoritative). Implementation: a small post-build scripts/finalize-mobile-build.mjs, or a Vite transformIndexHtml plugin — equivalent.

B-B11 · Delete redundant proxy delete

File: src/routes/api/[...path]/+server.ts

hooks.server.ts is the canonical proxy and handles JWT-stripping on public auth endpoints. The duplicate [...path]/+server.ts lacks that safety. Removing it keeps one source of truth.

08Verification

Build-time

  • cd anotame-web && bun run build exits 0 with adapter-node output (web).
  • cd anotame-web && bun run build:mobile exits 0 with adapter-static output (no index.js server in build/).
  • npx cap sync updates ios/ and android/ without errors.
  • mvn -pl identity-service compile (and the other three) succeeds with the env-driven CORS line.

Web (regression)

  • docker compose up --build — login flow unchanged; HttpOnly jwt cookie set; theme loads server-side (no FOUC); admin route blocked for EMPLOYEE.
  • UI locale switch updates <html lang="..."> on next navigation.

iOS

  • npx cap open ios, run in Xcode simulator.
  • Log in; verify tokenStore contains the JWT (Capacitor storage inspector or debug log).
  • Hit an authenticated endpoint; confirm Authorization: Bearer ... header in Charles/Proxyman.
  • Quit and reopen the app; session restores from Preferences.
  • Trigger camera-based garment intake on a test order; photo upload succeeds.
  • Trigger biometric login on a second open; Face ID prompt and successful re-auth.

Android

  • npx cap open android, run in emulator. Same six steps as iOS.

CORS

  • From the iOS simulator, hit /auth/me against the staging Railway backend — expect 200, not a CORS-blocked preflight.
  • Confirm staging's ANOTAME_CORS_ORIGINS includes capacitor://localhost.

Out of scope (deferred)

  • Push notifications.
  • Barcode / QR scanner.
  • Offline-first order capture with sync.
  • App Store / Play Store submission, code signing, provisioning profiles.
