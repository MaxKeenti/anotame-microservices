Backend Consolidation and Client Access Plan

Recommended path: fix authorization and branch correctness first, reduce measured runtime cost without changing ownership, and build the client portal as a separate public security boundary. Database and process consolidation remain optional, evidence-driven workstreams.

Summary

The completed invoice confirms that memory is the dominant cost; runtime measurements still have to identify safe reductions. Client access is blocked until staff and client authorization models, tenant scope, and customer identity are enforceable and covered by tests.

Decision Gates

Every optional phase has a measurable entry condition and an agreed stop condition. A lower resource limit or fewer services is not, by itself, proof of lower cost or acceptable reliability.

Required before changes

Cost and Payback

  • The completed $26.18 invoice and current per-service production usage are recorded; memory dominates both views.
  • The Pro plan and $20.00 included monthly usage are recorded; retain the invoice ledger for exact reconciliation.
  • Measure process RSS, heap use, CPU, restarts, and active/idle database connections.
  • Set a minimum monthly saving and maximum payback period before Phase 4 or 5.

Observability and Reliability

  • Structured request timing is active in staging; normalize routes, add correlation fields, and aggregate production windows before claiming route p95/p99.
  • Staging is the release-gate environment and follows Git branch staging.
  • Define SLOs for login, order list, order detail, payment, delivery, and client verification.
  • Define rollback triggers for OOMs, 5xx rates, login failures, DB waits, and cold-start latency.

Tenancy and Staff Scope

  • Choose the tenant boundary: establishment, branch, or both.
  • Decision recorded: multiple Branches may be active, while every authenticated staff session carries exactly one selected branch_id.
  • Define which admin operations are establishment-wide and which still require a branch predicate.

Customer Identity

  • Define whether one verified phone/email maps to exactly one customer within a tenant.
  • Define deduplication and account-linking behavior for existing customer records.
  • Choose email, SMS, or both and record provider cost, delivery SLO, and abuse controls.

Phase 4 reverses the current service-owned database ADR. Proceed only with a replacement ADR that records measured savings, the larger failure and recovery radius, the migration method, and the point after which rollback is no longer lossless.

Decision Implications

  • Memory is the dominant invoice and current estimated cost, so Phase 1 should measure JVM and process memory before considering topology changes.
  • The four APIs represent about 74% of the current production total, while all four databases represent about 17%. Phase 4 has a low absolute savings ceiling on this baseline.
  • The three business APIs remain the larger theoretical consolidation target, but only a measured merged prototype can establish savings after Phase 1.
  • If the current $21.23 estimate closes unchanged and no other billable lines accrue, only about $1.23 is above the Pro plan's $20.00 included-usage floor. Savings below the floor create headroom rather than a lower invoice.
  • The 2026-07-10 baseline had no HTTP logs. Phase 0 now emits structured access logs in staging, but production and a representative observation period remain outstanding.

Railway project boundary: keep production and staging as environments in this project. Railway private networking is isolated by both project and environment, so separate API projects would lose private service discovery. A telemetry collector, if one becomes necessary, should run per environment in this project or use an external OTLP endpoint; a third cross-project collector cannot privately connect to these services. See Railway private-network isolation and third-party observability options.

Baseline conclusion: prioritize observability and effective JVM configuration before topology changes. Do not use repository Dockerfile heap values, the invoice, or successful health checks as substitutes for runtime heap evidence, route SLOs, and a representative traffic period.

Cost

Runtime Cost Control

Keep all four backend services. Reduce actual resource consumption first, then use platform limits as guardrails around known-good values.

Entry status: ready for one-variable-at-a-time experiments in staging, but not for production tuning. Phase 0 production rollout, numeric SLOs, a representative traffic period, and detailed JVM/native-memory measurements remain open.

Confirmed Current State

  • Current production deployments use the service Dockerfiles: identity, catalog, and operations declare -Xmx256m, while sales declares -Xmx384m. Runtime MaxHeapSize, committed heap, metaspace, and native memory still require measurement.
  • JDBC pools use min-size=0 and maximum sizes of 2 or 3.
  • Background connection validation is disabled, private database references are configured, and all nine production services currently report successful deployments.
  • The completed invoice is $26.18 and its visible memory line is $25.25. Current production usage is $3.68, estimated at $21.23, including $20.88 estimated memory.

Remaining Actions

  • Inspect the running JVM command, max heap, committed heap, metaspace, and native memory before setting JAVA_TOOL_OPTIONS or replica limits.
  • Compare RSS, heap, metaspace, and native memory before lowering any JVM or replica limit.
  • Treat Railway replica limits as crash and spend guardrails, not direct savings.
  • Execute the HTTP measurement plan above: normalize staging access logs, deploy Phase 0 telemetry to production, aggregate it, and collect the representative baseline.
Evaluate Serverless per service. Railway enables sleeping for a whole service, not selected routes. Development proves that all nine services can reach a sleeping state, but not that production sleeping is safe. Confirm that outbound traffic does not prevent sleep and test first-request latency, possible wake-up 502 responses, retry behavior, and user experience.
Keep stateful services awake unless tested otherwise. Do not assume PostgreSQL or a service with persistent outbound traffic will sleep safely. Start with low-traffic stateless APIs whose cold-start SLO permits it.
Deploy one change at a time. Change heap, replica limit, pool, or Serverless separately so regressions and savings remain attributable.

Exit gate: the invoice or projected usage shows a real reduction, p95 and cold-start SLOs still pass, and no service has OOM, restart, or connection-wait regressions.

Authorization

Authorization and Data Scope

Replace broad @Authenticated protection with an explicit permission matrix. Enforce role and scope in controllers and in the repository query that selects or mutates the resource.

Domain and action Recommended actor Required data predicate
User list, user detail, create, update, delete ADMIN Current establishment or explicit platform-admin policy.
Own profile, credentials, locale Authenticated staff self; ADMIN override Principal user ID must equal target ID. Prefer /users/me.
Order create/read/update/status/deliver and payment entry ADMIN, EMPLOYEE Order branch must be inside the staff token scope.
Order delete and order audit history ADMIN Order branch or establishment scope still applies.
Customer create/search/read/update; customer delete Staff for daily work; ADMIN for delete Use the chosen tenant policy; do not expose customers across establishments.
Catalog, price-list, schedule, and establishment writes ADMIN Establishment or branch ownership where applicable.
Operational KPI; financial and customer KPI Staff for operational; ADMIN for financial/customer Every aggregation must include the same tenant and branch rules as source rows.
Client order status and history Tracking proof or client session only Exact customer and tenant match; never staff-wide access.
Default deny every route family. Staff endpoints require explicit @RolesAllowed. Public and client endpoints use separate policies; a valid client credential must be rejected by all staff endpoints.
Scope reads and mutations in SQL. Use methods such as findByIdAndBranch and include branch/tenant criteria in order lists, summaries, payments, customers, audit logs, and every KPI query.
Prevent identifier probing. For client resources, return the same not-found behavior whether an order does not exist or belongs to another customer. Do not reveal cross-tenant existence.

The role matrix above is the recommended starting policy. Confirm it with the business owner before implementation, especially employee access to status, delivery, payment, audit, and KPI data.

3

Client

Client Access Foundation

Implement the public boundary in sales-service first, where orders and customers are owned. Keep client credentials, endpoint policies, cookies, DTOs, and audit events separate from the staff dashboard model.

Proposed Endpoints

  • POST /client/order-status with ticket and dedicated tracking proof in the request body.
  • POST /client/session/code-requests with a generic 202 Accepted response.
  • POST /client/session/code-verifications to establish a client session.
  • GET /client/orders and GET /client/orders/{id} under that session.
  • DELETE /client/session to revoke the current session.

Minimal Client DTO

  • Ticket, public status, received date, committed/estimated ready date, and delivered date.
  • Garment and service display names.
  • Total, paid, or balance only when the product policy explicitly allows it.
  • No pickup code, notes, full customer object, payment method, price adjustments, internal IDs, or audit history.

Never reuse pickupCode for web lookup. It is a six-digit value and currently authorizes physical delivery. The ticket number is sequential and is not a secret.

Add a dedicated tracking credential. Generate at least 128 bits with a cryptographically secure generator, encode it for display or QR use, store only a hash, never log it, and support rotation/revocation. For QR links, keep the proof in the URL fragment and have the page submit it by POST with Referrer-Policy: no-referrer.
Introduce verified customer contacts. Store normalized phone/email, verification time, tenant, and customer link. A contact must resolve unambiguously or require an additional linking step; application-only uniqueness checks are insufficient.
Persist one-time challenges safely. Store a challenge hash, customer/contact link, expiry, failed-attempt count, consumed time, and audit metadata. Suggested defaults are 5-10 minute expiry, at most 5 attempts, and a 30-60 second resend cooldown.
Apply layered abuse controls. Rate-limit independently by IP, normalized contact, ticket, and challenge. Share counters across replicas, use generic responses and comparable timing, add escalation or CAPTCHA after thresholds, and redact secrets and contact data from logs.
Prefer an opaque client session. Issue a random 256-bit token in a separate client_session cookie with HttpOnly, Secure, SameSite=Lax, and a restricted path. Store only its hash with customer, tenant, expiry, and revocation time; validate Origin and CSRF protection on authenticated state-changing requests.
Define the JWT alternative completely. If client JWTs are required, use a dedicated key, issuer, audience, token type, short expiry, customer and tenant claims, and an explicit verifier. A jti is revocable only when a server-side session or denylist checks it.

Exit gate: a client credential cannot authenticate to any staff route, customer A cannot read customer B, invalid proofs do not disclose order/contact existence, and client responses contain no internal fields.

4

Database

Optional Database Consolidation

Use one PostgreSQL service and one database with identity, sales, catalog, and operations schemas only when the measured database baseline justifies the migration and larger recovery radius.

Bootstrap and Ownership

  • Run a controlled bootstrap that creates schemas, owners, roles, and shared extensions once.
  • Use one owner/migrator and one runtime role per bounded context.
  • Remove CREATE from PUBLIC on the public schema and database where appropriate.
  • Set default privileges so new tables and sequences remain usable only by the intended runtime role.

Runtime Isolation

  • Runtime roles receive only required DML and sequence privileges in their own schema.
  • Runtime services never use owners, migrators, or a shared superuser.
  • No cross-schema foreign keys unless ownership is intentionally changed by ADR.
  • Test that every runtime role is denied reads and writes in the other three schemas.
Make all schema configuration explicit. Set JDBC currentSchema, quarkus.hibernate-orm.database.default-schema, quarkus.flyway.schemas, and Flyway-specific credentials. A common history-table name is safe because each table lives in a different schema.
Separate bootstrap from service migrations. Current baselines install extensions. Move cluster/database bootstrap concerns out of per-service Flyway execution so schema migrators do not need broad privileges or race each other at startup.
Keep branch integration explicit. Prefer the Phase 0 operations API or local projection. If a shared-database view is deliberately chosen, expose only a narrow operations-owned view and grant identity SELECT on that view, not the operations schema.
Rehearse a maintenance-window cutover. Provision and bootstrap the target, run schema-aware migrations, freeze writes, perform tested logical exports/imports, validate row counts and sequence values, switch connection settings, then smoke-test every service.
State the rollback boundary. Keep old databases read-only during validation. A simple rollback is lossless only before writes begin on the new database; afterward use a forward fix or a rehearsed change-capture/replay procedure.
Test both recovery levels. Railway volume backup/PITR recovers the shared PostgreSQL service as a unit. Maintain logical per-schema exports or restore to a sibling service and extract one schema when bounded-context recovery is required.

Exit gate: the replacement ADR is approved; RPO and RTO are recorded; migration, role-denial, Flyway, whole-instance restore, schema extraction, and pre-write rollback drills all pass in staging.

5

Services

Optional Business-Service Consolidation

Merge sales, catalog, and operations into one deployable modular monolith only if Phase 1 leaves material JVM/container cost. Keep identity-service separate.

Required Structure

  • Keep domain, application, and infrastructure packages per bounded context.
  • Expose module operations through application ports rather than cross-module repository access.
  • Keep route namespaces, exception contracts, migrations, and schema ownership explicit.
  • Document whether the merger transfers data ownership or preserves only organizational boundaries.

Operational Controls

  • Do not place the staff JWT private signing key in the business process.
  • Retain separate client-session secrets and reject client credentials on staff routes.
  • Measure startup time, RSS, DB connections, and deployment blast radius before and after.
  • Keep the previous service deployments available for a rehearsed rollback.

Separate database credentials inside one process preserve configuration discipline but are not a security boundary against compromise of that process. Do not claim isolation that the merged runtime cannot provide.

6

Exceptional

Full Backend Consolidation

Treat one process for identity and all business domains as an exceptional tradeoff. Package separation does not isolate signing authority inside a compromised process.

Use an external identity or signing authority. A separate identity provider or signing service preserves authority isolation. KMS prevents private-key extraction, but a compromised application with signing permission may still request signatures.
Recalculate the actual benefit. If Phase 1 and Phase 5 already solve the cost problem, skip full consolidation. Simplicity claims must include deployment, recovery, and security blast radius.

Default decision: do not execute Phase 6 without a concrete business constraint that cannot be met by the earlier phases.

Launch Blockers

These conditions block client launch. The same authorization work should also be completed before merging business services.

Must resolve

Branch Source Is Invalid

  • Identity currently queries an operations-owned table through the identity datasource.
  • SQL errors are swallowed and replaced with a default branch.
  • Pre-Phase 0 production can select multiple assignments nondeterministically; the staged contract rejects ambiguity until the ADR 0007 branch chooser exists.

Staff Authorization Is Too Broad

  • Many controllers use only @Authenticated.
  • Order, customer, payment, KPI, catalog, operations, and some user routes lack a complete backend role policy.
  • Frontend visibility is not an authorization control.

Queries Lack Tenant Scope

  • Order reads, summaries, details, payments, audit logs, and KPI queries are not consistently branch-scoped.
  • Customer ownership across branches or establishments is not defined.
  • Mutation methods load resources by ID without a scope predicate.

Client Identity Does Not Exist

  • Customer phone/email fields have no database uniqueness or verification state.
  • The pickup code is short, generated for delivery, and must not become a web credential.
  • No challenge, client-session, abuse-control, or client-token verification model exists.

Release Environment Exists; Telemetry Baseline Is Open

  • Railway staging follows Git branch staging, is restored from the production backup, and passes the Phase 0 smoke tests.
  • Structured HTTP logs are available in staging, but production route error rates and p50/p95/p99 latency still lack a representative observation period.
  • Production now uses the service Dockerfiles and their heap flags, but runtime committed heap, metaspace, and native-memory evidence remains incomplete.

Recommended Sequence

This order fixes existing correctness issues, delivers public access on a defensible authorization model, and leaves consolidation as an optional response to measured cost.

Final order
  1. Phase 0: instrument and fix branch assignment. Remove the failing cross-database lookup and default fallback before relying on branch claims.
  2. Phase 1: tune measured runtime cost. Keep four services, test one resource change at a time, and verify service-level sleeping where appropriate.
  3. Phase 2: enforce the role and scope matrix. Make staff routes default-deny and add tenant/branch predicates to every affected query and mutation.
  4. Phase 3: build and verify client access. Add dedicated tracking proof, verified contacts, opaque client sessions, abuse controls, and minimal DTOs.
  5. Phase 4 only if database savings meet the gate. Replace the ADR, rehearse schema migration and recovery, then cut over during a controlled write freeze.
  6. Phase 5 only if JVM savings still meet the gate. Merge the three business services as explicit modules while keeping identity separate.
  7. Avoid Phase 6 by default. Full consolidation requires external identity authority and a business constraint that earlier phases cannot satisfy.
